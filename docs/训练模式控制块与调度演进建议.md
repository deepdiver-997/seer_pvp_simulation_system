# 训练模式控制块与调度演进建议

## 目的

这份文档用于记录当前 C/S 交互方式的取舍结论，回答三个问题：

1. 现在的训练模式交互是否足够可用。
2. 当前 `run_mutex` 和控制块设计的主要问题在哪里。
3. 后续如果要优化并发调度，推荐往什么方向演进。

当前结论是：

- 现阶段应优先保证 **FSM 逻辑正确**。
- 单一测试客户端联调时，当前方案 **可以继续使用**。
- 调度与并发优化先记账，不作为当前主线任务。

## 当前实现概览

当前训练模式的核心链路如下：

- 客户端通过单 socket 向服务端发送带 `uuid` 的消息。
- `TrainingControlBlock` 负责：
  - TCP 粘包拆包
  - 按 `uuid` 路由消息
  - 维护 `pending_inputs_`
  - 在 `wait_for_input()` 后继续唤醒对应 `BattleContext`
- `BattleContext` 保存：
  - `currentState`
  - `current_player_id_`
  - `m_buffer`
  - `is_empty`
- `BattleFsm::run()` 进入状态机主循环，并通过 `run_mutex` 保证同一个 context 串行推进。

这是一套“**事件驱动 + context 内串行**”的设计，只是训练模式因为是单 socket 多路复用，控制块部分显得更复杂。

## 当前方案为什么现在够用

### 1. 当前主要场景是单客户端联调

现在最常见的使用方式是：

- 一个训练客户端
- 一个对局
- 用 `SYNC_STATE` + `step`/`skill`/`pet` 手动推进

这个场景下：

- 网络吞吐不是瓶颈
- SQLite 只读不是瓶颈
- 真正的核心风险仍然是 FSM 时点、死亡结算、跳转条件、技能效果链是否正确

所以当前阶段把精力放在逻辑正确性上是合理的。

### 2. `run_mutex` 虽然不优雅，但很稳

`BattleFsm::run()` 入口加锁的直接收益是：

- 保证单个 `BattleContext` 不会被并发推进
- 避免多次 `post()` 抢同一个 context 造成状态破坏
- 对当前“逻辑仍在快速变化”的阶段非常友好

它的代价是：

- 线程池线程可能阻塞在锁上
- 同一个 context 被高频唤醒时会有额外调度开销

但在单训练客户端场景里，这个代价目前可以接受。

## 当前设计的复杂点

### 1. 训练模式控制块比真实对战模式更绕

训练模式当前是：

- 单 socket
- 多 `uuid`
- 一个连接里顺序提交双方输入
- 还要在“当前是否正处于等待输入状态”时决定是否立刻唤醒 FSM

因此引入了这些额外结构：

- `waiting_uuid_`
- `pending_inputs_`
- `ctx->m_buffer`
- `ctx->is_empty`

这些结构本身不是错误，只是说明训练模式的“单连接多路复用”天然更复杂。

### 2. `run_mutex` 是正确性优先的保守实现

问题不在“会不会错”，而在“够不够优雅”。

当前模式下：

- 有输入到来时会 `post(run)`
- 某些 `async_write` 完成后也会 `post(run)`
- 如果此时另一个线程已经在推进同一个 context，新线程会阻塞在 `run_mutex`

这不是逻辑错误，但不是长期最理想的调度方式。

### 3. 当前不适合做“全局轮询扫描所有 context”

一种直觉上的改法是：

- 控制块不直接唤醒 `run`
- 改成定时扫描所有 context
- 看哪些 context 满足“需要输入且已有输入”就继续跑

不推荐这样做，原因是：

- 当前系统本质上是事件驱动，不是 tick 驱动
- 全局扫描会额外引入周期性开销
- 语义也更模糊，不如“谁收到事件谁尝试唤醒自己”清晰

## 对客户端的建议

可以约定客户端也缓存最近一次 `SYNC_STATE`，并遵循以下策略：

- 只有 `needInput == true` 时才发送操作
- 只有 `inputPlayer` 对应自己时才发送该玩家的输入
- 只有 `needDeathSwitch` 为真时才发送死后切宠输入

这样做的好处：

- 减少无效请求
- 调试体验更清晰
- 更容易复现某个时点的真实上下文

但需要强调：

**客户端缓存只能作为发送优化，不能作为权威状态。**

服务端仍然必须校验：

- 当前是不是该时点
- 当前是不是该玩家输入
- 当前动作是否合法

因为客户端状态可能过期、乱序，或者被多个调试终端同时操作。

## 后续推荐的演进方向

如果未来要解决 `run_mutex` 的阻塞问题，推荐把调度模型改成：

### 每个 context 一个 mailbox + 原子调度位

核心思想：

- 每个 `BattleContext` 自己维护待处理输入队列
- 每个 `BattleContext` 自己维护少量原子状态，例如：
  - `scheduled`
  - `running`
  - `waiting_for_input`
- 任何事件到来时：
  - 先把数据写入该 context 的输入队列
  - 再尝试用 CAS 将 `scheduled` 从 `false` 置为 `true`
  - 成功后才 `post()` 一个真正的 worker

worker 执行时：

1. CAS 抢占 `running`
2. 若失败，直接返回，不阻塞线程池
3. 若成功，循环推进该 context，直到：
   - 需要输入但没有输入
   - 到达结束状态
   - 当前批次工作已经处理完
4. 退出前清理 `running/scheduled`
5. 如果退出过程中又来了新输入，再次尝试调度

这类设计的优点：

- 不需要 `run_mutex` 阻塞线程
- 不需要全局扫描 context
- 一个 context 在任意时刻仍然只会有一个真实执行者
- 更适合未来多 context 并发训练

## 为什么现在先不做

主要有三个原因：

### 1. 当前主矛盾不是吞吐，而是规则正确性

目前系统最重要的是：

- FSM 跳转是否正确
- 技能命中/效果失效/技能无效分支是否正确
- 先后手死亡结算是否正确
- 回合效果和未来触发器是否正确

这些问题优先级明显高于调度优化。

### 2. 当前训练模式仍以“人工联调”居多

只要还是：

- 单客户端
- 单局或少量局
- 边看 `SYNC_STATE` 边调

那么并发调度优化的收益不会特别大。

### 3. 过早重构调度层会增加排错难度

在 FSM 逻辑还在持续变动时，同时重构：

- 控制块
- 调度策略
- 原子状态机

会让问题来源更难定位。

## 长期建议

如果以后要做高频训练、自博弈、批量回放，建议区分两条通道：

### 1. 调试通道

保留当前 C/S 方式：

- TCP
- `INIT_BATTLE`
- `SELECT_SKILL / CHOOSE_PET`
- `SYNC_STATE`

适合：

- 命令行联调
- 前端演示
- 手动复盘

### 2. 训练通道

单独提供进程内接口，例如：

- `submit_action(ctx, player, action)`
- `run_until_need_input(ctx)`
- `snapshot(ctx)`

适合：

- 自博弈
- AI 批量训练
- 压测

这样可以避免训练时重复消耗：

- socket 通信
- 报文封包/解包
- JSON 状态同步
- 控制块多路复用逻辑

## 后续可选演进：进一步包装 IO 层

除了把训练模式调度改成 mailbox/actor 式之外，后期还可以把“消息来源”再抽一层。

核心思想是：

- `BattleFsm`
- `BattleContext`
- 技能/魂印/效果系统

这些核心战斗模块都不需要知道数据究竟来自：

- TCP socket
- 共享内存
- 进程内直接调用

它们只需要接收统一格式的“战斗输入事件”。

### 推荐分层

建议把 IO / 输入来源包装为统一接口，例如：

- `IInputBackend`
- `ITransport`
- `IBattleIngress`

名字可以再定，但职责应保持一致：

- 接收初始化请求
- 接收行动请求
- 接收状态同步请求
- 把它们转换为统一的 battle event，交给控制块或上下文

### 两类后端

后续可以并行支持两种后端：

#### 1. 在线/正常对战后端

保留当前路线：

- `Asio`
- `io_context`
- socket
- TCP 粘包拆包

适合：

- 在线对战
- 前端联调
- 命令行工具
- 远程客户端接入

#### 2. 专业训练后端

训练模式后端可以进一步做轻量化，例如：

- 共享内存读取
- 进程内无锁队列
- 直接函数调用 / API 入口

适合：

- 大规模自博弈
- 批量训练
- 高吞吐压测

这样可以减少：

- 一层 IO 线程池
- socket 缓冲和系统调用
- 封包/解包
- 文本/JSON 状态同步开销

### 编译与链接建议

这个方向可以结合编译开关，但不建议让大量业务代码直接依赖宏分叉。

更稳的方式是：

- 保留统一接口
- 各后端分别实现
- 用 CMake 选项或少量编译宏控制“是否编译/链接某个后端”

不推荐的方式：

- 在 `BattleFsm`、`BattleContext`、核心效果逻辑里大量写 `#ifdef TRAINING_MODE`

推荐的方式：

- `AsioBattleIngress`
- `SharedMemoryBattleIngress`
- `LocalTrainingIngress`

由工厂或启动配置决定实例化哪一个。

这样做的好处：

- 宏只停留在边界层
- 核心逻辑不被平台/后端细节污染
- 后期替换训练输入通道时，不容易影响 FSM 主链

## 分层草图

下面是一种推荐的长期结构草图：

```text
                    +----------------------+
                    |   Client / Trainer   |
                    +----------+-----------+
                               |
             +-----------------+-----------------+
             |                                   |
             v                                   v
   +----------------------+          +----------------------+
   |  AsioBattleIngress   |          | SharedMemoryIngress  |
   |  socket + io_context |          | shm / local queue    |
   +----------+-----------+          +----------+-----------+
              |                                 |
              +-----------------+---------------+
                                |
                                v
                    +--------------------------+
                    |      IControlBlock       |
                    | route / validate / queue |
                    +------------+-------------+
                                 |
                                 v
                    +--------------------------+
                    |      BattleContext       |
                    | input buffer / state     |
                    +------------+-------------+
                                 |
                                 v
                    +--------------------------+
                    |        BattleFsm         |
                    | run / state transition   |
                    +------------+-------------+
                                 |
                                 v
                    +--------------------------+
                    | Effects / Skills / Marks |
                    +--------------------------+
```

如果后期训练通道进一步内聚，还可以演进成：

```text
Trainer
  |
  v
LocalTrainingRunner
  |
  +-- submit_action(ctx, ...)
  +-- run_until_need_input(ctx)
  +-- snapshot(ctx)
  |
  v
BattleContext + BattleFsm + Effect System
```

这种结构的含义是：

- 在线对战继续走 C/S
- 高性能训练可以不经过 socket
- 但两者最终复用同一套战斗核心

## 当前阶段的明确决策

当前项目阶段采用以下策略：

1. 保留当前训练模式控制块设计，不优先重构并发调度。
2. 保留 `run_mutex` 作为单 context 串行推进保护。
3. 客户端可以缓存 `SYNC_STATE`，但服务端仍保持权威校验。
4. 当前优先级仍然是战斗逻辑、FSM 时点与效果系统正确性。
5. 等规则链路稳定后，再考虑把训练模式演进为 mailbox/actor 式调度。
6. 如果未来训练吞吐成为瓶颈，再把 IO 层进一步拆为在线后端与专业训练后端。

## 后续待办

以下内容暂记为后续优化方向：

- 给 `BattleContext` 增加非阻塞调度位设计草案
- 训练模式从 `run_mutex` 迁移到 mailbox/actor 模型
- 区分调试通道与训练通道
- 提供更轻量的进程内训练入口
- 设计统一的输入后端接口，支持 `Asio` 与共享内存/本地训练双实现
