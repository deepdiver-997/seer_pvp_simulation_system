# 命令行客户端使用说明

本文说明如何使用命令行客户端 `sim_training_cli` 进行训练模式联调和单步调试。

## 1. 编译

在仓库根目录执行：

```bash
cmake -S . -B build && cmake --build build -j
```

## 2. 启动服务端

```bash
./build/sim_server
```

如果希望看更详细的 FSM 追踪日志：

```bash
BATTLE_FSM_VERBOSE=1 ./build/sim_server
```

## 3. 启动命令行客户端

另开一个终端：

```bash
./build/sim_training_cli
```

启动后会看到 `cli>` 提示符。

## 4. 基础命令

| 命令 | 说明 |
|------|------|
| `help` | 查看命令帮助 |
| `use <uuid>` | 设置默认对局 uuid，后续命令可不再重复写 uuid |
| `quit` / `exit` | 退出客户端 |

## 5. 对战管理

- `init012 [uuid]`
: 使用内置 012 双方阵容初始化对局（6 只波克尔，双方携带不同技能组）。

## 6. 操作输入

- `skill <robot> <skill_idx> [uuid]`
: 提交技能操作。robot 为 0 或 1，skill_idx 为 0~4。

- `medicine <robot> <med_idx> [uuid]`
: 提交药剂操作。

- `pet <robot> <pet_idx> [uuid]`
: 提交换宠操作。pet_idx 为背包槽位 0~5。

- `turn <r0_action> <r0_idx> <r1_action> <r1_idx> [uuid]`
: 一次性提交双方回合操作，适合按回合推进。
: action 只能是 `skill`、`medicine`、`pet`。
: 执行后自动 sync 显示结果状态。

## 7. 运行模式控制

- `mode step [uuid]`
: 开启单步调试模式。此后 FSM 每执行一个状态就会停下，需要用 `debug step` 手动推进。

- `mode run [uuid]`
: 开启连续运行模式（默认）。FSM 会一直执行到需要输入或对战结束才停下。

## 8. 调试命令

- `debug step [uuid]`
: 单步执行当前状态，然后停在下一个状态。执行后自动 sync 显示新状态。

- `debug break <state_id> [uuid]`
: 切换指定状态的断点（toggle 模式）。再次执行同一 state_id 会取消断点。
: 例如 `debug break 5` 在"双方先手权结算"处打断点。

- `debug breaks [uuid]`
: 列出当前所有断点。

- `debug full [uuid]`
: 获取完整状态快照（含效果表、异常状态、工作区全部字段）。

## 9. 状态查看

- `sync [uuid]`
: 请求服务端 `SYNC_STATE`，查看当前状态快照 JSON。

- `status [uuid]`
: 基于 `SYNC_STATE` 输出摘要：当前时点、回合数、是否等待输入、是否需要死亡切换。

- `check`
: 查看双方当前场上精灵（名称、血量、技能与 PP）。

- `check <player>`
: 查看某一方背包 6 只精灵的名称和血量。例如 `check 0`。

- `check <player,slot>`
: 查看某一方某个背包位精灵的详细信息（hp、技能名、剩余 pp/最大 pp）。
: 例如 `check 0,1` 表示 0 号选手第 1 号位精灵。

## 10. 其他

- `heartbeat [uuid]`
: 发送心跳包。

- `raw <cmd> <uuid> [int1 int2 ...]`
: 发送自定义命令码和整型参数（网络字节序自动处理）。

- `poll [timeout_ms]`
: 主动拉取一次服务端返回（默认 200ms 超时）。

## 11. 推荐联调流程

### 连续运行模式（快速推进）

```
init012 1
mode run 1           # 确认为连续模式（默认就是）
turn skill 0 skill 0 1   # 双方都用技能0，自动运行到下次需要输入
status 1
```

### 单步调试模式（逐个时点验证）

```
init012 1
mode step 1          # 开启单步模式
debug break 14       # 在"先手方攻击伤害结算"处打断点
turn skill 0 skill 0 1   # 提交回合操作
debug step 1         # 执行第一个状态 GAME_START，停在下一个
debug step 1         # 执行下一个，继续停在再下一个
...                  # 逐步推进直到断点自动停住
debug full 1         # 在关键时点查看完整状态
debug break 14       # 取消断点
mode run 1           # 恢复连续运行
```

### 死后切换调试

```
status 1             # 看到 needDeathSwitch.player0=true
pet 0 3 1            # 玩家0切换到第3只精灵
debug step 1         # 单步观察死后切换流程
```

## 12. 输出说明

客户端输出带固定格式：

- `[HH:MM:SS] [SEND] ...` 发送了什么
- `[HH:MM:SS] [RECV] ...` 收到了什么
- `[HH:MM:SS] [INFO] ...` 一般信息
- `[HH:MM:SS] [WARN|ERROR] ...` 参数或网络异常

服务端输出带固定格式：

- `[YYYY-MM-DD HH:MM:SS] [LEVEL] [battle] ...` 战斗逻辑日志
- `[YYYY-MM-DD HH:MM:SS] [LEVEL] [control] ...` 控制块日志

方便从时间线快速对应请求与处理。
